# Context: Fix Electron Fork Bomb on desktop:package

## Requirements

### Goal
Running `desktop:package` and opening the resulting packaged app causes a fork bomb: Electron spawns another Electron, which spawns another, etc. — the machine freezes. The fix must ensure the packaged app never accidentally launches a new GUI Electron instance from within the process tree.

### Acceptance Criteria
- [ ] `npm run desktop:package` completes without spawning any GUI Electron instances during the build
- [ ] Opening the resulting `.app` starts exactly one GUI Electron window
- [ ] The Next.js standalone server and proxy start correctly (as Node.js processes, not as Electron GUI)
- [ ] No fork bomb when opening the packaged app, even if `server.js` or the proxy spawn internal child processes
- [ ] Dev mode (`npm run desktop`) continues to work unchanged

### Out of Scope
- Changing the Next.js build or standalone configuration
- Adding a separate bundled Node.js binary
- Any changes to the UI or app behavior

### Edge Cases
- `ELECTRON_RUN_AS_NODE` missing from a grandchild process env → should still not trigger GUI
- `requestSingleInstanceLock` returning true for a directly-spawned binary → should be prevented at env level
- Packaged app opened while dev mode is running → single instance lock should focus existing window

## Q&A Record
- Q: Does the fork bomb happen during `electron-builder --mac` or when opening the resulting `.app`?
  A: **Confirmed — when opening the `.app`** after desktop:package finishes.

## Codebase Analysis

### Root Cause

**Primary gap:** `ELECTRON_RUN_AS_NODE: "1"` is passed to immediate children (`startProdServer`, `startProxy`) but is NOT set on `process.env` of the main Electron process itself.

This means:
- Immediate children (server.js, claude-proxy.js) get `ELECTRON_RUN_AS_NODE: "1"` ✓
- BUT any grandchild spawned by server.js (e.g., Next.js 15 internal workers) that builds its own env object (`{ NODE_ENV: "production", ... }`) will NOT inherit `ELECTRON_RUN_AS_NODE` ✗
- A grandchild spawning `process.execPath` without `ELECTRON_RUN_AS_NODE` launches a FULL GUI Electron instance

**Secondary gap:** `requestSingleInstanceLock` is designed to prevent multiple `.app` GUI instances, but a directly-spawned Electron binary (not launched through the `.app` bundle) may resolve to a different user data directory than the main `.app`, bypassing the lock.

In `buildChildEnv` (electron/main.ts:183):
```typescript
return {
  ...process.env,  // ← process.env does NOT have ELECTRON_RUN_AS_NODE
  ...envFromFile,
  ELECTRON_RUN_AS_NODE: "1",  // ← only immediate child gets this
};
```

When `server.js` spawns workers (Next.js 15 may spawn workers for instrumentation, after(), etc.):
```javascript
// Inside Next.js worker spawning (hypothetical):
spawn(process.execPath, [workerScript], { env: { NODE_ENV: "production" } })
//                                               ^^ no ELECTRON_RUN_AS_NODE!
// → Electron binary launches as GUI → loads main.js → fork bomb
```

### Existing Patterns to Follow

- `buildChildEnv` at `electron/main.ts:183` — builds env for prod child processes
- `startProxy` env at `electron/main.ts:273` — inline env for proxy spawn
- `app.whenReady` at `electron/main.ts:381` — lifecycle entry point where fixes should be applied
- `requestSingleInstanceLock` at `electron/main.ts:369` — existing guard against multiple GUI instances

### Affected Files

- `electron/main.ts` (modify) — add `process.env.ELECTRON_RUN_AS_NODE = "1"` early in packaged mode, add secondary env-level guard in `buildChildEnv` (belt and suspenders)
- `electron/dist/main.js` (regenerate) — compiled output, regenerated by `desktop:build-main`

### The Fix

**In `app.whenReady()`, before any spawning:**
```typescript
if (app.isPackaged) {
  // Set on process.env itself so ALL descendant spawns inherit it,
  // including Next.js internal workers that build custom env objects from process.env.
  process.env.ELECTRON_RUN_AS_NODE = "1";
}
```

This is safe because:
- The flag is already checked at Electron's startup — setting it after startup has no effect on the current process
- It only affects spawned children and their descendants
- Dev mode (`app.isPackaged = false`) is untouched

The existing explicit `ELECTRON_RUN_AS_NODE: "1"` in `buildChildEnv` and `startProxy` stays as a second layer of defense.

### Risks

- **`process.env.ELECTRON_RUN_AS_NODE` set too early** (Low) — Could be read by some early Electron internal code. Mitigation: set it inside `app.whenReady()`, not at module level.
- **Dev mode regression** (Low) — `app.isPackaged` guard ensures dev mode is unaffected.
- **Next.js workers spawning via non-`process.env` path** (Low) — Edge case; the fix covers the primary inheritance path. The existing explicit flags in `buildChildEnv`/`startProxy` cover the rest.
- **`requestSingleInstanceLock` user-data-dir mismatch** (Med) — When a naked Electron binary (not via `.app`) is spawned, macOS may assign a different user data dir. The env fix prevents this scenario from occurring at all.
